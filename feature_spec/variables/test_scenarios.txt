Variables Feature — Local MCP Test Scenarios
============================================

These are concrete prompts to type into a Claude AI session connected to the local
keboola-mcp-server. Run them in order — each scenario builds on or cleans up after
the previous one. After each prompt, verify the "Expected" state described below it.

SETUP
-----
Make sure the local MCP server is running (see CLAUDE.md for .mcp.json setup).
All storage-side checks use `get_configs` with component_id="keboola.variables".

SCENARIO 0 — Baseline: no variables configs exist yet
══════════════════════════════════════════════════════════════════════════════
PROMPT:
  List all configurations for component "keboola.variables".

EXPECTED:
  - Empty list (or only pre-existing vars configs unrelated to our test).
  - Note down the count — after the test suite this count must be the same.

SCENARIO 1 — Create SQL transformation WITH variables and a default value
══════════════════════════════════════════════════════════════════════════════
PROMPT:
  Create a new SQL transformation called "Variables Test Transformation".
  Description: "Test transformation for variables feature".
  Add a single SQL code block named "Query" with the query: SELECT '{{env}}' AS environment.
  Attach two variables to this transformation:
    - name="env", type="string", default_value="production"
    - name="api_key", type="vault" (no default value)

EXPECTED:
  - Tool returns success=True and a version number (this must be the FINAL version
    after variables were linked, not the version from the initial create call).
  - A new "keboola.variables" config named
    "Variables definition for keboola.snowflake-transformation/<new_config_id>"
    appears when you list keboola.variables configs.
  - That vars config has configuration.variables = [
        {name: "env", type: "string"},
        {name: "api_key", type: "vault"}
    ]
  - It has a row named "Default Values" with values=[{name: "env", value: "production"}].
    (api_key has no default so it must NOT appear in the Default Values row.)
  - The parent transformation config has variables_id set to the vars config ID.

NOTE: Save the transformation configuration_id for use in later scenarios.

══════════════════════════════════════════════════════════════════════════════
KNOWN LIMITATIONS (out of scope per RFC)
══════════════════════════════════════════════════════════════════════════════

1. LISTING variables: there is no dedicated "list variables" tool. Use:
     get_configs(component_id="keboola.variables")
   This returns all vars configs in the project. To find the one linked to a
   specific parent, look for the name pattern:
     "Variables definition for {component_id}/{config_id}"
   Or check the parent config's variables_id field.

2. NAMED VALUE SETS: only the "Default Values" row is managed. Multiple named
   value sets (e.g. "staging values", "prod values") are not supported —
   you would need to create those manually via the Storage API.

3. ROW-BASED COMPONENTS: variables cannot be attached via add_config_row or
   update_config_row. Only root-level configs (create_config / update_config)
   and SQL transformations are supported.

4. VARS CONFIG LIFECYCLE: passing variables=[] deletes the linked keboola.variables
   config from Storage entirely (it is not merely cleared or orphaned). If you
   later re-add variables, a fresh vars config is created. To leave variables
   unchanged, omit the parameter (pass None/null) instead of passing [].
